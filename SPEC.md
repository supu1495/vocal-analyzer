## プロダクトの目的

カラオケ録音をAIで分析し、歌唱力向上のための具体的なフィードバックを提供する。
ボイストレーナーがいない環境でも、客観的な指標で自分の歌唱を振り返れるようにする。

### ターゲットユーザー

- カラオケを趣味として楽しむ人
- 歌唱力を向上させたい人
- ボイストレーニングの補助ツールとして使いたい人

### 開発の背景・既存システムへの問題意識

既存のカラオケ採点システムはMIDIデータとの波形一致を重視する「音ゲー」的なアプローチを取っており、**楽曲本来の表現や音楽的な歌唱クオリティを正しく評価できていない**。このプロダクトはその課題を解決するために構築する。

**既存システムの具体的な問題点（Anti-Patterns）:**

- **ハードウェア依存** — マイク感度やゲイン設定によってスコアが変動する。キャリブレーション等による正規化が必要
- **「棒読み」の過大評価** — 機械的な音程維持（棒読み）が高評価される。楽曲の文脈に沿わない不自然な安定は評価を下げるべき
- **表現技法の回数主義** — 「しゃくり」「こぶし」等を回数だけで加点評価している。フレーズ内での密度や適切さで判断すべき

## コア評価ロジックの設計方針

### 基本思想

楽曲ごとの「理想的な歌唱モデル」との比較ではなく、**歌唱そのものの音楽的整合性を判別する汎用ロジック**を構築する。単純な加点方式ではなく、各要素をプロットした上で最終的な「歌唱特性」を結論づける。

### A. スコアマトリクス（多角的評価）

**UI表示方針：** 総合スコア1つをメインで表示。詳細ボタンで各軸の内訳を確認できる。

| 評価軸 | 内容 | 加点 / 減点 |
|---|---|---|
| 基本忠実度 | 音程・リズムの正確性 | ベースライン（加点） |
| 表現の不自然さ | 不自然に連続する技法、抑揚のないロングトーン（棒読み） | 減点要素 |
| 音楽的ダイナミクス | ブレスのタイミング、声の質感の変化（A/Bメロ・サビ対比は将来対応） | 加点要素 |

**棒読み検出の方針（Approach A）：**
- フレーズ内のピッチ分散が極端に小さい
- かつ技法検出数がゼロ
- → 「不自然な安定」として減点。実際にテスト・検証しながら閾値を調整する

### B. 技法密度分析（Technique Density Analysis）

1小節またはフレーズ単位で技法の出現回数をカウントし、一定の閾値を超えた場合は「表現の過剰」として評価を調整する。技法を「使っているかどうか」ではなく「適切な密度で使えているか」で評価する。

### C. テキストベースのフィードバック

抽象的な点数だけでなく、分析結果に基づいた具体的なテキストを出力する。出力形式：

- **歌唱特性** — 分析から導き出されたその人の歌い方の傾向
- **改善点** — 具体的に直すべきポイント
- **伸ばすべきポイント** — 現時点での強みと磨き方

**実装方針：**
ルールベーステンプレートで実装する（コスト0・安定動作）。フィードバック生成を独立した関数として切り出し、将来LLMに差し替えられる設計にする。LLM導入はAPIコストが許容できるタイミングで検討。

### 抽出する音響特徴量

| 特徴量 | ライブラリ | 備考 |
|---|---|---|
| ピッチ（F0） | Crepe | フレームごとの検出・信頼度付き |
| フォルマント | librosa | 声質・母音の分析に使用 |
| エネルギー（音量） | librosa | ダイナミクス・ブレス解析に使用 |
| BPM・リズム | librosa | テンポ検出・タイミングのズレ計測 |
| 歌唱技法 | 独自実装 | ビブラート・こぶし・フォール・しゃくり・ロングトーン |

## 非機能要件

### プライバシー・著作権

- **音声ファイルは一切保存しない**（著作権保護のため）
- 分析結果（数値・JSON）のみPostgreSQLに保存
- アップロードされた一時ファイルは分析完了後に即時削除

### セキュリティ

- JWT認証はhttpOnly Cookieで管理（XSSによるトークン窃取を防止）
- パスワードはbcryptでハッシュ化
- ログイン失敗のロックアウトはRedisで管理
- エラーメッセージはユーザー列挙攻撃を防ぐため曖昧化（「メールアドレスまたはパスワードが正しくありません」）
- 本番環境ではHTTPS必須

### パフォーマンス

- Demucsは処理が重いため非同期化（Celery）が前提
- Crepeはピッチ検出精度優先（librosa.pyinへの変更はしない）

## 設計上の決定事項

### analysis_results.user_id を Nullable のまま維持する理由

現状は`NOT NULL`化せずNullableを維持する。
将来的に「ログインなしで分析できる機能」（ゲスト分析・友人の分析代行など）を実装する予定があるため、NOT NULL化はそのタイミングまで保留。

### Demucsのスタブ化

CPUでの処理が遅すぎるため、Celery非同期処理の実装後に本番復帰させる。
現在は`audio/separator.py`がlibrosaで音声をそのままボーカルとして返すスタブ実装。

### CrepeをTensorFlowで動かす

ピッチ検出の精度重視のためlibrosa.pyinへの変更はしない。
hmmlearnとpybind11の競合を回避するため、Dockerfileで`crepe`を`--no-deps`でインストールする。

### JWTブラックリストを後回しにする理由

httpOnly CookieによりXSSでのトークン窃取リスクは軽減済み。
完全なセキュリティ対策（ログアウト後の即時無効化）は本番デプロイ前のタイミングで実装する。

## AI活用の方針

このプロダクトはClaude Codeを活用して開発している。以下は開発を通じて大切にする原則。

| | 原則 | 具体的な行動 |
|---|---|---|
| 🔍 | **自信満々な回答ほど疑う** | 数値・固有名詞・APIは必ず公式ドキュメントで確認する。AIは「たぶん」という感覚でも断言することがある |
| ✂️ | **一度に大きな作業をさせない** | 「全部やって」より「まず調べて→次に提案→最後に実装」と段階を分ける |
| 📎 | **コンテキストを惜しまず渡す** | エラー文・実際のコード・バージョン情報を渡す。「なんかうまくいかない」はNG |
| ↩️ | **おかしいと思ったら即突っ込む** | 「本当にそのメソッドある？」と聞き返すだけで正直に訂正してくれることが多い |
| ⏸️ | **承認フローをサボらない** | Claude Codeの確認ステップは面倒でも飛ばさない。AIの暴走を防ぐブレーキ |
| 🧠 | **「答えを出す機械」として使わない** | 一緒に考えるパートナーとして使う。最終判断は常に自分がする |
| ⚖️ | **設計レビューは丁寧に、実装は一気に出させる** | 設計・方針の確認は段階を踏んで慎重に行い、合意が取れた実装は一気に書かせる。速さと安全のバランスとして最も現実的なアプローチ |
